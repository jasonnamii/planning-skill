# Planning Skill

범용 기획 파이프라인. 형의 기획 사고 흐름을 6 Phase로 프로토콜화한 오케스트레이터.

이 스킬의 핵심 역할: **기획의 "언제·무엇을" 판단**하고, **"어떻게"는 전문 스킬/트리거에 위임**한다.

---

## 파이프라인 총괄

```
P0 상황배경 → P1 가설적 목표 → P2 가설확장+리서치 → P3 논리적 수렴 → P4 창의적 점프 → P5 기획안 진화
                                                         ↓
                                                   (정책기획 등: P5 직행)
```

| Phase | 이름 | 핵심 행위 | 호출 도구 | 상세 |
|-------|------|-----------|-----------|------|
| P0 | 상황배경 | 좌표계 설정 (의뢰자·배경·제약·성공기준) | 자체 | `references/phase0-setup.md` |
| P1 | 가설적 목표 | "완전한 X" MECE 분해 → 갭 매트릭스 | 절대자 트리거 | `references/phase1-hypothesis.md` |
| P2 | 가설확장+리서치 | 갭→리서치 가설→축별 조사→교차분석 | **research-frame** 스킬 + 연역수렴 | `references/phase2-research.md` |
| P3 | 논리적 수렴 | 스파인 도출 → 전략 방향 확정 → 기획안 뼈대 | 백본·스켈레톤 트리거 | `references/phase3-convergence.md` |
| P4 | 창의적 점프 | 논리적 결론에 점프기술 적용 → 복수 컨셉 | 맥가이버·아날로지·모순기술 등 | `references/phase4-jump.md` |
| P5 | 기획안 진화 | v1→v2→v3 반복. 형 지정 포맷으로 최종화 | SHE·제출청소·이쁘니 트리거 | `references/phase5-evolution.md` |

---

## 게이트 (Phase 간 통과 조건)

게이트를 두는 이유: 리서치 없이 기획하거나, 스파인 없이 축 나열하는 실패를 구조적으로 차단한다.

| 전환 | 통과 조건 | 미충족 시 |
|------|-----------|-----------|
| P0→P1 | 배경·의뢰자·제약·성공기준 중 3/4 확보 | 부족 항목 질문 |
| P1→P2 | 갭 매트릭스 완성 + 🔴 1개 이상 | 갭 없으면 "기획 불필요" 보고 |
| P2→P3 | 축별 최소 1건 교차 확인 + 확신도 50↑ | 추가 리서치 축 제안 |
| P3→P4 | 스파인 확정 + 형 방향 승인 | 스파인 후보 2~3개 → 형 선택 |
| P3→P5 | 형 "여기서 마무리" 또는 수렴 완료 판단 | P4 스킵, P5 직행 |
| P4→P5 | 점프 결과 형 확인 | 다른 점프기술 적용 또는 P3 복귀 |

## 포맷 규칙

기획안 최종 포맷은 **형이 직접 지정**한다. 스킬이 임의로 정하지 않는다.

| 항목 | 기본값 | 형 지정 시 |
|------|--------|-----------|
| 파일 형식 | Obsidian .md | 형이 명시한 포맷 우선 (.docx, .html 등) |
| 구조·스타일 | deliverable-engine 모래시계+블록설계 | 형 지정 스타일 체계 (HTML div, apple-design 등) |
| 줄 수·부록 | deliverable-engine 기본 | 형 지정값 (예: 40줄, 부록접기) |

형이 지정하지 않은 항목만 deliverable-engine 기본값 적용.

## 스킬 의존성 (Cascade)

이 스킬이 Phase별로 호출하는 외부 스킬·트리거. cascade-map §2 권장선행에 등록됨.

| Phase | 호출 대상 | 유형 | 호출 방식 |
|-------|----------|------|-----------|
| P1 | 절대자 | 트리거 (trigger-dictionary) | "완전한 X" MECE 분해 |
| P2 | **research-frame** | 스킬 (직접 호출) | 갭→리서치 가설→축별 조사 |
| P2 | 연역수렴 | 트리거 (trigger-dictionary) | 수렴검증 4축 적용 |
| P3 | 백본·스켈레톤 | 트리거 (trigger-dictionary) | 스파인 도출→기획안 뼈대 |
| P4 | **hit-skill** | 스킬 (직접 호출) | 자극설계 모드로 히트패턴 적용 |
| P4 | 맥가이버·아날로지·모순기술·넛지 | 트리거 (trigger-dictionary) | 점프기술 (형 승인 후) |
| P5 | SHE·제출청소·이쁘니 | 트리거 (trigger-dictionary) | 기획안 진화·최종화 |

**요약:** 스킬 직접 호출 2건(research-frame, hit-skill) + trigger-dictionary 경유 다수. planning-skill은 허브이고, 실제 "어떻게"는 전부 위임한다.

## 라우팅

이 스킬이 처리하지 않는 영역과 위임 대상. 대화 초반에 도메인을 감지하여 라우팅한다.

| 신호 | 위임 대상 | 이유 |
|------|-----------|------|
| 지자체·선거·정책·공약 | policy-planning | 행정단위 프로파일링, ABCD축, 제출청소 등 도메인 특화 프로토콜 보유 |
| 청사진→로드맵→액션 전환 | ceo-pipeline | 샤워효과 캐스케이드 특화. **로드맵·액션리스트 단독 요청 = ceo-pipeline 직행. "기획 후 로드맵까지" = planning-skill P0~P3 → P3 완료 시 ceo-pipeline 핸드오프** |
| 사업계획서·IR·피치덱 | bp-guide | VC 프레임 특화 |
| UI·화면설계·PRD | ui-action-designer | 피그마→PRD→Action 특화 |
| 리서치만 단독 수행 | research-frame | WHY→HOW→SCOPE 단독 실행 |
| 재무모델·매출추정 | financial-model | 가정→산출→시나리오 특화 |

**경계 상황:** "기획인데 정책 요소도 있다" → 플래닝 스킬로 시작, P2에서 policy-planning의 축별리서치 참조 가능. 핵심은 **메인 파이프라인을 누가 관장하느냐**로 판단.

## P4 점프기술 추천 로직

P3 수렴 완료 후, 형이 직접 지정하지 않으면 상황에 따라 추천한다.

| 상황 | 추천 트리거 | 이유 |
|------|------------|------|
| 타 분야 영감 필요 | 아날로지 | 구조적 유사성 전이 |
| 자원·시간 제약 내 돌파 | 맥가이버 | 제약 조건 내 재조합 |
| 양립 불가 요구 공존 | 모순기술 | 모순 해소 프레임 |
| 인간 반응·행동 설계 | hit-skill (자극설계 모드) | 인간 메커니즘 레이어 |
| 행동 유도·선택설계 | 넛지 | 선택 아키텍처 |
| 형이 직접 지정 | 해당 트리거 | — |

추천은 1~2개. 형 승인 후 trigger-dictionary 스킬 호출로 실행.

## planning-recipe 메타패턴 흡수

기존 planning-recipe에서 검증된 7가지 법칙 중 6개를 각 Phase에 내장했다.

| 메타패턴 | 반영 위치 | 방식 |
|----------|-----------|------|
| 절대자 프레임→역추적 | P1 | 절대자 트리거 호출 |
| MECE 분해→병렬 리서치 | P1→P2 | 갭 매트릭스가 리서치 축을 자동 정의 |
| 리서치→결정 3단 전환 | P2→P3 | 팩트→결정→기획반영 테이블 |
| 기획안 반복 진화 | P5 | v1→v2→v3 프로토콜 |
| 동일 리서치→복수 컨셉 | P4 | 프레이밍 변수(수혜자·가치축·제약수용·톤) |
| 갭 분석이 구조를 만든다 | P1 | 핵심 원리 |

"의뢰자 프로파일링 기반 일관성"은 P0 좌표계에서 의뢰자 프로파일을 필수 항목으로 설정하여 간접 반영.

## Gotchas

- **리서치 없이 기획 착수:** P1 갭 매트릭스만으로 바로 기획안 작성 시도. 반드시 P2를 거쳐야 "왜 이 방향인가"의 근거가 생긴다.
- **스파인 미확정 상태로 P4 진입:** 점프기술은 수렴된 결론 위에 적용해야 효과. 스파인 없이 점프하면 발산만 됨.
- **P4를 매번 강제:** 형이 "여기서 마무리"라고 하면 P3→P5 직행. 점프가 항상 필요한 건 아니다.
- **포맷 임의 결정:** .md 외 포맷이나 스타일 체계는 반드시 형이 지정. 스킬이 "HTML div 적용하겠습니다" 같은 자의적 판단 금지.
- **도메인 라우팅 무시:** 정책기획 신호가 명확한데 플래닝 스킬로 강행하면 ABCD축·제출청소 등 도메인 프로토콜 누락.
- **P0 과잉 세팅:** 프로젝트 CLAUDE.md가 이미 있으면 P0 대부분 스킵 가능. 매번 처음부터 좌표계를 세우지 않는다.
- **QC 결과를 산출물에 포함:** P5의 기획언어 금지패턴 체크, 수치 정합성 검증 등은 내부 검증 절차다. 산출물에 "금지패턴 준수 확인" 테이블이나 "사용 안 함" 목록을 넣으면 금지 표현이 산출물에 노출된다. QC는 작성 과정에서 수행하고, 그 결과(체크리스트·검증 로그)는 산출물에서 제거한다.
- **라우팅 위임 시 대상 스킬 내부 구조 인용:** 위임을 제안할 때 대상 스킬의 Phase 번호, 단계명, 내부 프로토콜명 등을 산출물에 인용하지 않는다. "정책기획 전문 파이프라인이 더 적합할 수 있다" 수준으로 안내하고, 형이 상세를 원하면 해당 스킬 호출로 대응한다.
